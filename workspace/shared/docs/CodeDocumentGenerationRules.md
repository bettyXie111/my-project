# 软著四份产物生成与综合检查规则

> 用途：为每个需求的四份交付产物提供统一的、可执行的生成与检查规范。
> 适用范围：`{A}` 需求目录下的 `源代码`、`操作手册`、`源代码文档`、`软件著作权登记申请表`。
> 目标：让 AI 在生成、整理、校验和修复时只输出“可定位、可验证、可落地”的内容。

## 1. 执行目标

AI 在处理软著材料时，必须同时满足以下目标：

1. 产物完整：四份产物及其衍生文件全部落盘。
2. 内容真实：所有描述都能在代码、配置、页面或截图中定位。
3. 三重一致：名称、版本、功能范围在各产物之间一致。
4. 可审查：材料不含模板污染、AI 套话、无关敏感信息和明显占位内容。
5. 可修复：一旦失败，能够明确定位失败项、原因和修复动作。

## 2. 输入与输出约定

### 2.1 输入

执行器至少应获得以下输入：

- `{A}`：软件全称，用于目录名与文件名前缀。
- `{PROJECT_ROOT}`：工作区根目录。
- `{OUTPUT_DIR}`：当前需求目录，默认 `{PROJECT_ROOT}/{A}`。
- `代码根目录`：用于定位实现证据的源码目录。
- `系统运行界面`：用于截图和界面说明校验。

### 2.2 必须输出的产物

以下内容缺一不可：

- `{OUTPUT_DIR}/{A}操作手册.md`
- `{OUTPUT_DIR}/{A}操作手册.docx`
- `{OUTPUT_DIR}/{A}操作手册.pdf`
- `{OUTPUT_DIR}/{A}源代码文档.md`
- `{OUTPUT_DIR}/{A}源代码文档.docx`
- `{OUTPUT_DIR}/{A}源代码文档.pdf`
- `{OUTPUT_DIR}/软件著作权登记申请表.md`
- `{OUTPUT_DIR}/软件著作权登记申请表.docx`
- `{OUTPUT_DIR}/images/`
- `{OUTPUT_DIR}/apps/`、`{OUTPUT_DIR}/scripts/` 或其他实际代码目录

说明：

- “源代码”指需求目录下实际提交与运行的代码，不单独要求再生成一个纯说明文件。
- “综合检查”必须覆盖四类对象：实际源代码、操作手册、源代码文档、申请表。

## 3. 总体执行原则

### 3.1 真实优先

- 文档中每个功能描述、页面名称、按钮名称、字段名称、规则说明，必须能在代码、配置、路由、截图或运行界面中定位。
- 无法定位的内容一律不得输出。
- 不允许为了凑字数加入通用行业背景、空泛价值描述或不存在的流程。

### 3.2 一致优先

以下字段必须在四类对象中保持一致：

- 软件全称
- 版本号
- 核心功能模块名称
- 页面/菜单/按钮/字段的业务命名
- 关键业务流程与约束

### 3.3 证据优先

AI 生成的每一段功能性描述，都应能回答以下问题：

1. 这段话对应哪个模块或页面？
2. 它对应的代码路径是什么？
3. 它在代码中的函数、类、接口、路由、配置项或字段是什么？
4. 它在截图中体现在哪个界面元素上？

不能回答以上问题的段落，视为无效内容。

## 4. 四类对象的生成与检查规则

## 4.1 源代码

### 4.1.1 目标

源代码必须体现真实业务实现，且具备独立性、可读性和可抽样提交性。

### 4.1.2 强制规则

- 代码目录中必须存在可运行或可解释的业务代码，禁止仅保留空壳结构。
- 核心业务模块必须可定位，例如：
  - 页面路由
  - 接口定义
  - 数据模型
  - 表单字段
  - 业务服务
  - 校验逻辑
- 代码中不得保留明显的脚手架示例、演示残留、无关测试样本、无意义占位页面。
- 禁止用仅修改变量名、注释、空格、换行的方式伪装原创。

### 4.1.3 结构级改写要求

若代码来源存在模板痕迹，必须做结构级调整，至少覆盖以下动作中的多项：

1. 拆分或合并函数，改变原始层级。
2. 调整模块组织方式，避免目录结构完全模板化。
3. 重写条件分支、数据处理和调用顺序，但不改变业务结果。
4. 引入业务相关的校验器、格式化器、适配器或工具函数。
5. 删除示例页、演示接口、无用常量、无意义注释堆。
6. 用业务命名替换泛化命名，例如 `item/list/data/manage`。

### 4.1.4 源代码检查项

必须检查：

- 是否存在真实业务模块。
- 是否存在明显模板污染。
- 是否存在敏感信息泄露。
- 是否存在大段无关框架代码挤占首尾页。
- 是否能为手册和申请表中的功能描述提供定位证据。

## 4.2 操作手册

### 4.2.1 目标

操作手册用于说明真实系统如何被操作，必须以实际界面和真实流程为准。

### 4.2.2 强制规则

- 必须基于真实运行界面截图，不得使用占位图、示意图或与系统不一致的图。
- 每个核心模块至少要有页面说明或流程说明。
- 每张截图附近必须有图题，格式为 `图X-Y 描述`。
- 每个主要功能章节都应描述：
  - 页面或弹窗名称
  - 关键字段至少 3 个
  - 关键按钮至少 1 个
  - 至少 1 条真实业务规则或校验提示
  - 对应代码定位证据

### 4.2.3 必备交互覆盖

若系统存在数据维护功能，则手册必须覆盖可见交互：

- 新增
- 编辑
- 删除或删除确认
- 查看或详情
- 至少 2 个弹窗、抽屉或确认交互

若某模块不具备以上动作，必须以该模块的真实等价动作替代，并写明触发结果。

### 4.2.4 文风要求

- 禁止出现“功能强大、界面简洁、操作方便、系统先进”等空泛套话。
- 禁止使用大量“首先、其次、最后、综上所述”等模板连接词。
- 必须使用真实业务词汇、字段名称、状态流转和校验规则。
- 必须删除交付元信息章节，如：
  - 版本信息
  - 交付一致性检查清单
  - 截图复核要点

### 4.2.5 操作手册检查项

必须检查：

- 章节骨架是否完整。
- 截图是否真实存在且被引用。
- 图题格式是否合规。
- 页面、字段、按钮、规则是否能在代码或界面中定位。
- 字数是否位于 3000 到 8000 字之间。
- 是否存在模板化空话、幻觉描述、重复段落。

## 4.3 源代码文档

### 4.3.1 目标

源代码文档是提交审查的代码抽样文档，不是功能说明书，也不是 PRD。

### 4.3.2 强制规则

- 内容应以代码行为主体，不得混入大段行业背景、业务介绍、宣传文案或 PRD 段落。
- 文档中的代码行必须来源于实际代码目录。
- 禁止插入与代码无关的自然语言充数。
- 禁止出现可见占位符，例如 `[padding]`。
- 默认目标为 3000 行、60 页、每页 50 行；若执行口径变化，以当前任务明确要求为准。

### 4.3.3 代码采样规则

生成器必须按稳定顺序拼接代码流：

1. 项目入口
2. 路由与页面
3. 业务模块
4. 数据模型与服务
5. 配置与工具
6. 其余代码按路径字典序

采样策略：

- 总行数大于 3000：取整体代码流前 1500 行和后 1500 行。
- 总行数小于等于 3000：全部收录；不足部分只允许使用真实代码行循环补齐，不得使用空行补齐。

### 4.3.4 过滤规则配置

- 公共过滤与跳过规则的唯一配置源为 `E:\copyRight\workspace\shared\config\source_doc_rules.json`。
- `shared/scripts/build_materials.py` 只负责按配置执行，不得在实现层维护分叉规则。
- 项目级补充过滤仅允许追加，不得削弱公共门禁。

### 4.3.5 源代码文档检查项

必须检查：

- 代码来源是否真实。
- 是否包含行业叙述、运营文案、PRD 章节词汇或无关说明。
- 是否满足行数、分页和非空补齐要求。
- 是否包含敏感路径、作者信息、时间戳、邮箱、外链等内容。

## 4.4 软件著作权登记申请表

### 4.4.1 目标

申请表用于概括软件基本信息与功能范围，必须与代码和手册严格一致。

### 4.4.2 强制规则

- 软件全称必须与操作手册、源代码文档一致。
- 版本号必须一致，格式固定为 `V1.0` 或任务另行指定的同类格式。
- 软件分类统一为实际类型，例如 `应用软件`。
- 源程序量必须与源代码文档总行数一致。
- 开发完成日期必须存在，格式固定 `YYYY-MM-DD`。
- 无法从项目或用户输入可靠推断的主体信息允许留空，但不得编造。

### 4.4.3 功能描述要求

申请表中的功能摘要必须：

- 覆盖真实核心模块。
- 与手册章节可映射。
- 与代码模块可定位。
- 不得出现文档中没有、代码中也没有的能力描述。

### 4.4.4 申请表检查项

必须检查：

- 名称、版本、日期、源程序量是否正确。
- 功能摘要是否与手册和代码一致。
- 是否包含虚构模块、虚构接口、虚构对接对象。

## 5. 跨产物一致性规则

以下规则必须 100% 通过：

### 5.1 名称一致

- 四类对象中的软件全称必须完全一致。
- 不允许一处带编号、一处不带编号；一处简称、一处全称。

### 5.2 版本一致

- 四类对象中的版本号必须完全一致。
- `V` 必须大写。

### 5.3 模块一致

- 手册章节中的模块名称必须能在代码中命中至少一处。
- 申请表中的功能描述必须能映射到手册章节和代码模块。
- 代码中的核心业务模块必须在手册或申请表中有对应说明。

### 5.4 功能一致

- 手册写到的功能，代码里必须存在实现或可验证入口。
- 申请表写到的功能，手册里必须有操作说明或功能说明。
- 不允许任何一份文档独自声明其他产物没有的核心能力。

## 6. 证据写法规范

为保证 AI 输出可验证，每个主要功能点都应包含“代码定位证据”行。

推荐格式：

```md
代码定位证据：apps/api/routers/sample.py；SampleCreateRequest；/samples/create；样本新增弹窗
```

证据来源至少命中以下之一：

- 文件路径
- 类名
- 函数名
- 路由名
- 接口路径
- 表单 schema
- 菜单项
- 配置键

规则：

- 证据必须真实可搜索。
- 证据必须指向当前需求的代码，而不是共享脚本或历史目录。
- 一个功能点至少给出 1 行证据；复杂流程建议给出多项证据。

## 7. 敏感信息与卫生规则

所有交付物必须移除或避免以下信息：

- 外网地址
- 本机真实路径
- 用户目录
- 作者姓名
- 邮箱
- 时间戳
- 无意义注释堆
- 演示、测试、占位、截图专用假数据说明

允许保留的例外：

- 申请表中的主体信息字段，在用户明确提供时可保留。
- 确属业务字段的日期、编号、状态，不视为敏感泄露。

## 8. AI 生成约束

AI 在生成四份产物时必须遵守以下约束：

1. 只写能定位到代码、截图或配置的内容。
2. 每个核心功能点必须附带定位证据。
3. 优先使用业务字段、业务动作、业务规则，不使用空泛评价词。
4. 不得臆造第三方对接、权限体系、统计口径、页面入口、异常流程。
5. 不得把 PRD、运行日志、交付说明直接复制进手册或源代码文档。

## 9. 推荐执行顺序

1. 整理并确认代码目录。
2. 抽取真实模块、页面、字段、按钮和规则。
3. 生成操作手册并补充截图与证据。
4. 生成源代码文档并执行纯度过滤。
5. 生成申请表并对齐名称、版本、行数和功能摘要。
6. 执行综合校验。
7. 若失败，按失败项逐条修复后重新校验。

## 10. 最终通过标准

只有当以下条件全部满足时，才可视为通过：

1. 四类对象全部存在。
2. 名称、版本、功能范围一致。
3. 操作手册截图真实、章节完整、说明可定位。
4. 源代码文档纯度合格，无无关文案污染。
5. 申请表字段真实且与其他产物对齐。
6. 代码中存在与文档描述对应的实现证据。
7. 没有敏感信息泄露、模板污染、AI 套话和明显幻觉描述。
