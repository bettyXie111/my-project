# 软著四份产物自动校验与失败判定规则

> 用途：定义自动校验器对每个需求产物应如何检查、何时判定失败、失败后如何修复。
> 适用对象：`源代码`、`操作手册`、`源代码文档`、`软件著作权登记申请表`。
> 目标：让 AI、脚本和人工复核都使用同一套失败标准。

## 1. 校验范围

自动校验必须覆盖以下对象：

- 需求目录下的实际源代码
- `{A}操作手册.md/docx/pdf`
- `{A}源代码文档.md/docx/pdf`
- `软件著作权登记申请表.md/docx`
- `images/` 截图目录

说明：

- 若某文件按流程应生成但不存在，直接判定失败。
- 若文件存在但内容明显不合规，不得视为通过。

## 2. 校验输入

执行器至少需要以下输入：

- `{PROJECT_ROOT}`
- `{A}`：软件全称
- `材料目录`：通常为 `{PROJECT_ROOT}/{A}`
- `代码根目录`
- 是否要求同时检查 `docx/pdf`

## 3. 校验结果级别

自动校验结果只分为两类：

- `PASS`：所有强制项通过。
- `FAIL`：任一强制项失败。

可附带辅助级别：

- `WARN`：当前不阻断，但应在本轮修复。

原则：

- 一票否决项命中时，直接 `FAIL`。
- 一致性、真实性、定位证据、敏感信息、文件缺失均属于强制项。

## 4. 一票否决项

命中以下任一项，直接判定 `FAIL`：

1. 缺少必交付文件。
2. 文档描述了代码中不存在的功能、页面、按钮、字段、接口或流程。
3. 软件名称或版本号在各产物之间不一致。
4. 文档中出现明显模板套话，且无法提供真实业务细节支撑。
5. 文档或代码材料泄露外网地址、真实路径、作者信息、邮箱或无关时间戳。
6. 源代码文档大面积混入行业文案、PRD 段落或非代码叙述。
7. 操作手册截图不是实际系统截图，或截图与说明不匹配。
8. 申请表中的功能摘要与代码、手册无法映射。
9. 代码定位证据缺失，或证据无法在代码中命中。

## 5. 文件存在性检查

### 5.1 必查文件

必须存在：

- `{A}/{A}操作手册.md`
- `{A}/{A}源代码文档.md`
- `{A}/软件著作权登记申请表.md`

若启用完整交付检查，还必须存在：

- `{A}/{A}操作手册.docx`
- `{A}/{A}操作手册.pdf`
- `{A}/{A}源代码文档.docx`
- `{A}/{A}源代码文档.pdf`
- `{A}/软件著作权登记申请表.docx`
- `{A}/images/`

### 5.2 失败判定

任一文件不存在，直接 `FAIL`。

## 6. 一致性检查

## 6.1 名称一致性

检查对象：

- 操作手册
- 源代码文档
- 申请表

通过条件：

- 三份文档中提取出的软件全称完全一致。

失败条件：

- 名称提取失败。
- 名称不同。
- 一处使用简称、一处使用全称、一处带编号一处不带编号。

## 6.2 版本一致性

通过条件：

- 三份文档中版本号完全一致。
- 版本格式符合 `V1.0` 这类大写 `V` 形式。

失败条件：

- 版本提取失败。
- 版本不一致。
- 存在 `v1.0`、`1.0` 等格式偏差。

## 6.3 模块与功能一致性

通过条件：

- 手册中的每个核心模块都能在代码中命中至少一处。
- 申请表中的功能摘要可映射到手册章节和代码模块。
- 代码中的核心业务模块在手册或申请表中有对应说明。

失败条件：

- 文档说到的模块无法在代码中定位。
- 代码存在明显核心模块，但手册和申请表完全未提及。
- 申请表功能描述为泛化表述，无法映射到真实实现。

## 7. 真实性与反幻觉检查

## 7.1 文档真实性

通过条件：

- 文档中的页面、字段、按钮、流程、规则均能在代码、截图或运行界面中找到依据。

失败条件：

- 描述了不存在的按钮、菜单、弹窗、接口、字段或流程分支。
- 描述了不存在的第三方对接、权限体系、算法能力或统计口径。

## 7.2 代码定位证据

通过条件：

- 每个主要功能点至少有一条 `代码定位证据`。
- 证据中的路径、类名、函数名、路由名、接口路径或字段名能在代码目录中检索命中。

失败条件：

- 缺少证据行。
- 证据为空、过短或无可检索实体。
- 证据中的 token 在代码目录中完全找不到。

## 8. 模板污染与 AI 痕迹检查

## 8.1 文本模板污染

重点检查以下表达：

- 功能强大
- 界面简洁
- 操作简单
- 系统先进
- 首先、其次、最后、综上所述

通过条件：

- 文档以业务字段、业务流程、业务规则为主体。

失败条件：

- 出现成段空话。
- 出现重复句式、重复段落、固定模板节奏。
- 无法从段落中提取真实字段、页面、规则信息。

## 8.2 代码模板污染

通过条件：

- 代码中不存在大段与业务无关的示例、演示、脚手架残留。
- 核心实现体现业务命名和业务逻辑。

失败条件：

- 首尾页主要由框架示例代码组成。
- 登录、分页、上传、增删改查等关键模块完全照搬模板结构且无业务化改写。

## 9. 截图与操作手册专项检查

通过条件：

- `images/` 目录存在且截图文件可用。
- 手册中的图片引用全部有效。
- 每张图附近存在图题，格式类似 `图3-2 描述`。
- 主要页面说明包含字段、按钮和规则信息。
- 操作手册字数位于 3000 到 8000 字之间。
- 至少体现真实交互入口和至少 2 处弹窗或确认操作。

失败条件：

- 图片缺失或引用失效。
- 只有静态首页截图，没有可操作页面。
- 图题缺失。
- 截图说明与界面元素对不上。
- 操作手册字数少于 3000 或多于 8000。

## 10. 源代码文档专项检查

通过条件：

- 文档内容以代码为主体。
- 不含行业背景、PRD 章节、宣传文案、交付说明等无关内容。
- 不含占位符、空白补齐、无意义说明段。
- 满足公共行数与分页要求，或满足当前任务显式要求。

失败条件：

- 命中公共纯度黑名单关键词。
- 出现可见占位符。
- 出现大量非代码自然语言段落。
- 行数、分页或补齐方式不符合规则。

## 11. 申请表专项检查

通过条件：

- 软件全称、版本号、源程序量、开发完成日期均存在且正确。
- 功能摘要与手册、代码可映射。
- 主体信息未编造。

失败条件：

- 字段缺失或格式错误。
- 源程序量与源代码文档不一致。
- 功能摘要与实际实现脱节。
- 填入无法确认的主体信息。

## 12. 敏感信息检查

必须检查以下模式：

- `http://`、`https://`、`www.`
- Windows 盘符路径，例如 `C:\`、`D:\`、`E:\`
- `\Users\`
- `author`
- `created at`
- `updated at`
- 邮箱地址

失败条件：

- 任一模式在交付文档中命中，直接 `FAIL`。

例外：

- 申请表中用户明确提供的主体信息不视为敏感泄露。

## 13. 自动修复优先顺序

自动修复必须按以下顺序进行，避免无效返工：

1. 文件缺失
2. 名称与版本不一致
3. 功能幻觉与证据缺失
4. 截图与手册不一致
5. 源代码文档纯度问题
6. 申请表字段与源程序量问题
7. 模板空话与 AI 痕迹
8. 代码结构模板污染

## 14. 失败输出格式

自动校验器输出失败时，必须至少给出：

- `失败级别`：ERROR 或 WARN
- `失败对象`：代码 / 操作手册 / 源代码文档 / 申请表
- `失败文件`
- `失败行号`（能给出时必须给出）
- `失败原因`
- `修复动作`

推荐格式：

```text
[ERROR] E:\copyRight\workspace\项目A\项目A操作手册.md:128
原因：代码定位证据缺失，当前章节描述了“样本审核弹窗”，但未提供路径或路由证据。
修复：补充“代码定位证据：apps/web/pages/review.vue；openReviewDialog；/review/detail”。
```

## 15. 最终判定标准

满足以下全部条件时，自动校验结果才可输出 `PASS`：

1. 所有必交付文件存在。
2. 名称、版本、模块、功能描述一致。
3. 手册描述全部可在代码、截图或界面中定位。
4. 申请表字段真实且与其他产物对齐。
5. 源代码文档纯度合格。
6. 没有敏感信息、模板空话、占位内容和幻觉描述。
7. 代码定位证据可搜索命中。

只要其中任一项失败，最终结果必须为 `FAIL`。
