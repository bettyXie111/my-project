# VR多模态交互媒介内容创作与发布系统需求规格说明书

## 一、产品概述

“VR多模态交互媒介内容创作与发布系统”面向高校传媒/数字艺术实验室、文旅展陈公司、企业培训内容团队与独立创作者，提供从“素材管理→场景搭建→交互编排→多端预览→版本发布→数据回收”的一体化闭环。系统强调低门槛创作（可视化编排）、可追溯交付（版本与资源一致性）、可复用资产（模板与组件库）与可运营发布（渠道、权限、统计）。

本版本以“Web 端创作与发布管理”为主，VR 端运行包/播放器不在本版本实现范围内，但需要在数据模型与发布流程中预留“运行端适配信息”（例如启动参数、资源清单、交互图谱导出）。

## 二、技术栈（用户确认：使用建议）

- 前端框架/运行时：React + Vite + TypeScript
- 后端框架/运行时：FastAPI + Python 3.11
- 数据库/存储：SQLite（默认）
- 认证/身份方案：JWT + RBAC（本地账号体系）
- 部署目标：Windows 本地运行
- CI/CD 工具：暂不配置（在本 PRD 说明中明确）
- 测试栈：pytest（后端）+ Playwright（可选 E2E）

说明：本项目以可交付为优先，暂不配置远程 CI/CD；若后续需要上云或持续集成，可在保持接口与目录结构不变的前提下补充 GitHub Actions 与容器化脚本。

## 行业背景

VR/沉浸式互动内容的生产链条往往割裂：资产在网盘/IM 中流转，场景搭建在不同工具里完成，交互规则依赖工程同学手工写脚本，版本发布靠口头约定与“最后一次打包”。这会带来三个高频问题：

1. **资源一致性难保障**：同一场景多版本素材混用、缺失依赖、路径不一致导致运行端报错。
2. **交互规则难复用**：触发器、状态机、热区、时间线等逻辑分散在个人工程文件，无法沉淀为团队资产。
3. **交付审计与回溯困难**：客户验收的是某一次演示包，但无法快速定位对应的资源清单、变更记录与责任人。

因此需要一个“面向 VR 互动内容”的轻量级生产与发布管理系统：用统一的资源与场景模型表达内容，用可视化编排表达交互，用版本与发布记录保证交付一致性，并用权限与统计支撑多团队协作。

## 用户角色

- **内容策划/编导**：定义项目目标、场景结构、叙事节奏与交互节点；查看预览与版本差异；发起发布申请。
- **美术/建模/动效**：上传与维护模型、贴图、音视频与特效资源；对资源做标签与规范化；处理缺失依赖与替换。
- **交互设计/技术美术**：在可视化编辑器中配置热区、触发条件、状态与反馈；维护组件模板与交互预设。
- **项目经理/交付负责人**：管理里程碑、版本、发布渠道与交付包；导出资源清单、变更记录与验收说明。
- **审核/管理员**：管理用户、组织、角色权限；配置发布渠道；审计操作日志与发布记录；处理回滚与冻结。

## 核心业务流程

1. **创建项目**：填写项目名称、客户/课程信息、目标设备（如 PCVR/一体机/展陈主机）与分辨率/帧率约束；创建默认场景与资源目录。
2. **资源入库与规范化**：上传模型/贴图/音视频/字幕等；系统计算 hash、生成预览图与元数据；进行“命名规范/格式/大小”校验并给出提示。
3. **场景搭建**：在场景树中创建节点（空间、道具、UI 面板、音源、触发器容器）；绑定资源引用；维护层级与定位信息（本版本以数据结构与列表编辑为主，预留 3D 视口字段）。
4. **交互编排**：通过“事件—条件—动作（ECA）”或简化状态机配置交互：
   - 事件：进入区域/点击/凝视/手势/语音命令/时间到达/变量变化
   - 条件：变量判断、权限判断、资源就绪、冷却时间
   - 动作：播放动画/音频、切换场景、显示 UI、写入变量、触发脚本导出
5. **多端预览（Web 预览）**：生成“交互图谱预览”和“资源依赖清单”，用于评审与验收前检查；支持按版本对比差异（新增/删除/替换资源、交互规则变更）。
6. **版本打标与冻结**：将当前项目快照生成版本（v1.0.0 等），冻结资源清单与交互规则；生成可追溯的变更记录。
7. **发布与回滚**：选择发布渠道（本地导出包/内网下载/二维码链接预留）、填写发布说明与目标设备信息；发布后可回滚到历史版本并重新发布。
8. **交付审计**：导出“版本资源清单（含 hash）”“交互规则摘要”“发布记录”“操作审计日志”；用于甲方验收或课程归档。

## 数据模型

核心实体（SQLite 存储，字段简化）：

- Project（项目）：id, name, customer, target_device, constraints_json, status, created_at
- Asset（资源）：id, project_id, type(model/texture/audio/video/ui), filename, sha256, size, tags, meta_json, preview_path, created_at
- Scene（场景）：id, project_id, name, root_node_id, created_at
- SceneNode（场景节点）：id, scene_id, parent_id, name, node_type, transform_json, asset_ref_id(nullable), props_json
- Interaction（交互规则）：id, project_id, scene_id, name, enabled, graph_json（事件/条件/动作结构）, created_at
- Variable（变量）：id, project_id, key, value_type, default_value, scope(project/scene), updated_at
- Version（版本）：id, project_id, version, snapshot_json（场景/交互/变量引用快照）, asset_manifest_json（资源清单+hash）, created_at
- PublishChannel（发布渠道）：id, name, type(local_export/intranet_placeholder), config_json, enabled
- PublishRecord（发布记录）：id, project_id, version_id, channel_id, note, artifact_path, created_at
- User / Role / Permission：RBAC（菜单、接口与数据范围）
- AuditLog（审计日志）：id, actor_id, action, target_type, target_id, detail_json, created_at

一致性约束与索引建议：
- Asset 按 (project_id, sha256) 去重；Version 生成时写入 asset_manifest_json，保证交付可追溯。
- Interaction.graph_json 需做 schema 校验（事件/条件/动作枚举与参数），防止运行端导出失败。

## 功能列表

### 1）项目与协作
- 项目创建/编辑/归档；项目成员与角色分配（RBAC）。
- 项目概览页：当前版本、资源数量、场景数量、交互数量、最近发布记录。

### 2）资源管理（资产库）
- 上传资源（拖拽/批量），自动计算 hash 与基础元数据提取（类型、大小、时长等可选）。
- 资源列表：筛选（类型/标签/使用状态）、预览、下载、替换、删除（引用检查）。
- 资源规范检查：命名规则、重复资源提示、过大文件提示、缺失贴图/依赖提示（最小实现以规则提示为主）。

### 3）场景管理
- 场景列表与场景树编辑（新增/删除/重命名/拖拽排序）。
- 节点属性编辑：基础属性（名称、类型）与 props_json（如碰撞体/可交互标记/提示文案）。
- 资源引用：将 Asset 绑定到节点，支持查看“被哪些节点引用”。

### 4）交互编排（可视化规则）
- 交互规则列表：启用/禁用、复制、版本对比。
- 规则编辑器（表单化）：选择事件→配置条件→配置动作；生成 graph_json。
- 交互预览：展示图谱摘要（节点数、触发链路）与潜在问题（循环、未引用变量、缺失动作参数）。

### 5）版本与发布
- 版本生成：输入版本号与说明，生成项目快照与资源清单（含 hash）。
- 版本对比：对比资源变更、场景节点变更与交互规则变更摘要。
- 发布：选择渠道、填写说明、生成导出目录（本地导出为主）；支持回滚发布。

### 6）运维与审计
- 用户/角色/权限管理；基础系统配置（发布渠道、资源目录）。
- 审计日志：按用户/时间/动作过滤；导出 CSV。

## 边界约束

- 本版本不提供真实 3D 视口编辑与 VR 运行端播放器，仅提供可导出的结构化数据与发布管理闭环。
- 资源解析与依赖分析以“可解释、可交付”为主：优先做规则提示与清单导出，不引入重型渲染与转换管线。
- 交互规则以 ECA/轻量状态机表达，避免复杂脚本执行引擎；导出格式以 JSON 为主，预留运行端适配字段。
- 默认本地目录存储资源；SQLite 作为单机存储满足交付演示与材料生成；不做多机并发锁优化。

## 验收标准

1. 可完成“项目→资源入库→场景搭建（树结构）→交互编排→版本生成→发布记录”的闭环，页面与接口无崩溃。
2. 资源上传后可在列表中检索、预览（至少显示元数据与文件信息）、并可查看引用关系；替换资源不破坏引用。
3. 交互编排能生成合法 graph_json，并能在预览页看到摘要与问题提示（至少能提示缺失参数/未引用变量）。
4. 版本生成后，能够导出包含 hash 的资源清单（manifest）与交互规则摘要；版本对比能展示变更摘要。
5. 发布记录可追溯：记录版本、渠道、说明与导出目录；支持从历史版本执行回滚发布。
6. RBAC 生效：不同角色对项目/资源/发布操作权限不同；审计日志能记录关键动作（上传、删除、版本、发布、回滚）。
7. 具备基础自动化验证：至少 1 组接口或页面冒烟测试通过（pytest；可选 Playwright）。

## 视觉风格关键词

- 未来感、沉浸式、偏深色但可读性强（深色底 + 高对比强调色）。
- 面向“创作工具”的专业控制台风格：左侧资源与场景导航、右侧属性面板、居中工作区；信息密度中高但层级清晰。
- 关键状态（草稿/冻结/已发布/回滚）使用统一状态色与徽标；危险操作（删除/回滚）有明显二次确认。
- 图标偏线性几何风格，按钮与表单对齐规整；支持快捷搜索与批量操作。
