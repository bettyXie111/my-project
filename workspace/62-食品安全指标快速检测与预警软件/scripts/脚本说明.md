## 2. 脚本生成过程

### 2.1 `generate_web_system.py`
- 生成来源：当前需求目录中的 wrapper 调用公共引擎生成。
- 生成目的：固定当前需求的业务上下文与调用方式，并作为 stage-2 的需求侧入口。
- 保留原则：不承载公共规则，仅保留当前需求专属调用信息；其中自动目录示例数据必须使用“样本记录/检测样本”等领域词，不得再出现“通用记录”。

### 2.2 `generate_prd.py`
- 生成来源：当前需求目录中的 wrapper 直接写入当前需求 PRD。
- 生成目的：把当前需求的行业背景、用户角色、核心流程、数据模型和验收口径固定在需求目录中，避免公共 PRD 生成器承载具体业务语义。
- 保留原则：不把食品安全业务细节回写到公共 `workspace/shared/scripts/generate_prd.py`，公共脚本只保留通用骨架能力。

### 2.3 `build_materials.py`
- 生成来源：公共材料生成引擎在当前需求目录的执行入口。
- 生成目的：在当前需求下重新抓图、更新操作手册、源代码文档和申请表。
- 保留原则：作为当前需求材料重建入口，不写入业务语义。

## 3. 脚本调用过程

### 3.1 常规流水线调用
- Stage-2：
  - 先运行当前需求目录下的 `scripts/generate_web_system.py`
  - 再执行基础测试
- Stage-4：
  - 先清空 `images/`
  - 再抓取当前需求截图
  - 再生成 `md/docx/pdf`

### 3.2 手动调用示例

```powershell
Set-Location -LiteralPath "E:\copyRight\workspace\62-食品安全指标快速检测与预警软件"
python -X utf8 .\scripts\generate_web_system.py --project-dir . --force
python -X utf8 .\scripts\build_materials.py --name "食品安全指标快速检测与预警软件" --output-dir . --pdf-engine libreoffice
```

## 4. 放置原则

- 只要脚本包含当前需求名、页面名、业务模块名或截图计划，就应放在当前需求目录下。
- 公共目录只保留通用引擎、规则和门禁。
- 当前需求目录 `scripts/` 只放当前需求的业务说明、专用入口和专用计划文件。

## 5. 公共脚本 / 需求脚本对照

| 类型 | 公共目录保留内容 | 当前需求目录内容 |
|---|---|---|
| 截图引擎 | `scripts/capture_screenshots_local.mjs` | `scripts/screenshot_plan.template.json` 或项目专用截图计划文件 |
| Web 生成 | `scripts/generate_web_system.py` 的通用骨架能力 | 当前需求的 `scripts/generate_web_system.py` 业务说明 |
| 材料生成 | `scripts/build_materials.py` | 本目录不重复实现，只保留调用说明 |
| 计划说明 | `workspace/shared/scripts/screenshot_plan.template.json` | 具体需求可复制后改成自己的截图计划 |

### 5.1 调用建议
- 公共脚本通过参数或环境变量接收需求计划。
- 当前需求只保留计划文件、业务说明和调用示例，不复制公共引擎。
- 若后续出现新的项目专用截图顺序，可在需求目录单独放 `screenshot_plan.json` 并通过 `SCREENSHOT_PLAN_FILE` 引用。
