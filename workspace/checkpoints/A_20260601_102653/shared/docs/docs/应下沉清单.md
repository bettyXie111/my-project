# 应下沉清单

> 口径：一旦脚本或文档开始承载“当前需求专属业务语义”，就应优先放到具体需求目录，而不是长期留在 `workspace/shared/scripts/` 或公共文档里。
> 判定标准：只要出现需求名、业务模块名、菜单名、路由名、示例数据、截图计划或按钮文案，就应评估下沉。

## 2. 需要下沉的内容特征

以下特征一旦出现在脚本中，优先考虑下沉：

- 出现具体需求名称、行业术语、页面标题、路由名或字段名。
- 出现项目专用截图计划、页面选择器、菜单结构、按钮文案。
- 直接引用某个需求的 PRD、手册章节、申请表字段或业务文案。
- 只服务单个项目的数据种子、页面生成、截图采集或打印检查。
- 只服务单个项目的 PRD 业务内容生成、行业背景与验收口径。

## 3. 拆分建议

如果脚本一部分是公共引擎、一部分是项目计划，建议拆成两层：

1. **公共层**：保留通用抓图、转换、校验、重试、时间戳处理。
2. **需求层**：保存截图计划 JSON、路由配置、字段配置、业务文案。

## 4. 迁移顺序建议

1. 先迁移“固定业务文案”和“固定页面计划”。
2. 再迁移“项目专用示例数据”和“项目专用截图脚本”。
3. 最后收口公共脚本里的默认值，确保公共目录只剩通用引擎。
