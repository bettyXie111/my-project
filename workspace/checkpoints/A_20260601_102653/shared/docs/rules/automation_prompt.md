【核心原则】
1. 内容独立
- 每个需求生成的 PRD、系统、三件套都必须围绕该需求的业务场景量身定制。
- 禁止套用通用业务模板。不同需求下的业务术语、数据模型、界面结构、截图内容必须显著不同。

2. 代码低重复
- 不同项目之间的代码重复率不得超过 30%。
- 禁止直接复用完整模块、整页 UI、整段接口实现或大段同构代码。
- 允许通过改变技术栈组合、目录结构、模块边界、命名方式、校验逻辑、注释风格来降低重复率。

3. 文字去 AI 化
- 避免常见 AI 套话，例如“可以非常方便地”“综上所述”“首先其次最后”。
- 使用自然、简洁、符合行业习惯的专业表达。
- 同一项目内 PRD、操作手册、申请表的语言风格必须统一；不同项目之间可根据行业特点调整风格。

4. UI 风格差异化
- 不同需求必须匹配不同的视觉风格关键词，并落实到实际系统界面。
- 主色调、布局、字体、圆角、图标风格、图表样式必须随需求主题变化。
- 系统截图必须体现这种差异化，禁止所有项目共用同一套 UI 模板。
- 操作手册截图数量不做固定，但必须根据实际功能覆盖酌情生成，且不少于 8 张。
- 截图生成稳定性与时间戳规则（避免反复误判“不是最新”）：
  - 截图生成脚本应在写入截图前先删除同名旧文件，禁止“覆盖写入同名文件但保留创建时间”的行为。
  - Windows 资源管理器“日期”列可能显示为“创建日期（CreationTime）”，即使内容已更新也会显示旧时间；因此截图脚本完成后应将 `CreationTime` 对齐到 `LastWriteTime`，或在验收时以命令行查询 `LastWriteTime` 为准。
  - 必须提供截图“去重复”门禁：至少执行一次字节级哈希（SHA256）去重；若业务允许，增加感知哈希（aHash/dHash）或“关键区域截图”策略，避免画面本质一致的截图混入材料。
  - 每次进行软著材料重建前，`build_materials.py` 必须先清空目标项目的 `images/` 目录，再重新抓取截图；禁止保留旧图后仅覆盖部分文件，避免不同需求的截图残留混用。

- 代码是否完成核心功能、具备测试数据、通过自动化验证。
- 代码、文字、界面是否完成去 AI 化处理。
- 软著三件套是否与系统实现一致。
- 不同需求项目之间是否保持内容独立、风格差异化与低重复率。
