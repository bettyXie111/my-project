# 软著材料规则补充（Web 系统）：自动校验规则与失败判定

> 用途：配合 `chinese-copyright-application` skill 的“阶段 C 一致性收束/最终验收”使用。
> 本文定义“必须检查什么、怎么判 Fail、Fail 后怎么改”。

## 0. 校验输入与目录约定
- 约定项目材料目录：`{A}/`
- 必须存在的文件（至少）：
  - `{A}/{A}操作手册.md`
  - `{A}/{A}源代码文档.md`
  - `{A}/软件著作权登记申请表.md`
  - 如已生成则必须存在：`docx/pdf` 对应文件与 `{A}/images/`

## 1. 一票否决（任一命中直接 Fail）
- 文档出现代码中不存在的功能/模块/页面/按钮/字段/接口/流程分支（幻觉描述）。
- 操作手册/申请表/源代码文档的软件名称或版本号不一致。
- 文档包含明显模板套话且缺少业务细节（如“功能强大、操作简单、界面友好”等成段出现）。
- 提交材料泄露外网地址、真实本机路径、作者信息、时间戳等敏感识别信息。
- 代码材料首尾页/重点抽取段落几乎全为框架/脚手架/示例代码（缺少自研业务逻辑）。

## 2. 三重一致性校验（必须 100% 通过）
- 名称一致：三件套均出现相同 `软件全称`。
- 版本一致：三件套均出现相同 `版本号`（格式如 `V1.0`，`V` 大写）。
- 模块一致：
  - 操作手册模块名必须能在代码中命中至少一处（目录名/类名/路由/接口/菜单项均可）。
  - 申请表的功能描述项必须能映射到手册章节与代码实现点。
- 功能一致：
  - 手册每个功能点都必须有“代码定位证据”（路径/类/函数/接口/路由至少一项）。
  - 代码中的核心业务模块必须在手册中存在对应说明（章节/小节）。

## 3. 文档语义雷同与 AI 痕迹校验（句子/段落级）
- 结构雷同：目录/章节标题与常见模板一致度过高时，要求重排章节结构。
- 句式雷同：连续段落出现高度相同句式或固定节奏，要求改写并引入项目细节。
- 空泛表达：出现“简洁/高效/强大/先进”等无落地信息，要求删除或替换为字段/规则/异常。
- 逻辑断层：流程步骤跳跃、缺少前置条件/权限/输入输出描述，要求补齐并给出代码证据。

## 4. 代码相似与结构级重复风险校验（结构/逻辑/算法块）
- 目录结构/命名组织：与通用模板过于一致时要求调整组织方式。
- 控制流与调用链：仅变量替换不视为原创性改写，要求结构级重构。
- 核心代码块：登录/权限/分页/上传等模块必须按业务重写，不得照搬通用模板实现。
- 开源冗余：删示例、删无用注释堆、删测试样例残留，避免“开源指纹大段重叠”。

## 5. 代码与文档卫生校验（敏感信息与格式）
- 禁止外网地址：`http://`、`https://`、`www.` 等（如确需保留必须脱敏且与业务无关内容移除）。
- 禁止真实路径：Windows 盘符路径（`C:\\`/`D:\\`/`E:\\`）、用户目录、CI 路径等。
- 禁止作者/时间戳：`author`、`created at`、具体人名/邮箱等（除申请表主体信息外）。
- 禁止凑字数：大量重复段落、无意义注释堆砌。

## 6. Fail 后修复策略（最小改动优先）
- 一致性问题：先修“名称/版本/模块术语”，再修功能映射，最后修文风与结构。
- 幻觉描述：以代码为准，删或改写为真实实现，并补上定位证据。
- AI 痕迹：删模板段，补业务细节（字段/规则/异常），并打散句式结构。
- 代码风险：按“10 个结构级手法”重构，避免仅做表面替换。
