# 软著材料规则补充（Web 系统）：源代码与申请材料文档生成规则

> 用途：作为 `chinese-copyright-application` skill 的规则补充文档使用。
> 适用范围：Web 系统（前后端一体或前后端分离均可）。
> 注意：官方不公开查重/比对算法细节；本文把你提供的“实务口径/风控阈值”落地为内部执行约束与验收红线，不作为公开法规条文引用。

## 0. 目标与总则（强制）

### 0.1 审查侧重点（执行口径）
- 源代码：做 `查重 + 原创性比对（核心）`，重点在 `结构 + 逻辑 + 算法块`，不是字符串替换。
- 说明书/操作手册：不做“全文相似度百分比查重”，但做：
  - `语义相似度比对`（对既有软著文档/AI 生成文本库）
  - `与源代码的一致性校验（重点）`
  - `AI 生成特征识别`（模板化、过规整、逻辑断层、幻觉描述）

### 0.2 三重一致性铁律（必须 100% 通过）
申请表 ↔ 源代码 ↔ 说明书/操作手册必须一致：
- 软件名称、版本号：完全一致（含 `V` 必须大写等格式要求）。
- 功能模块名：文档模块名要能对应到代码中的包名/目录/类名/路由/接口等可定位实体。
- 功能范围：文档写到的功能，代码里必须存在实现；代码里的核心模块，文档里必须有说明。
- 申请表日期字段：`开发完成日期` 必须存在，格式固定为 `YYYY-MM-DD`，取生成时的系统日期（`date.today().isoformat()`）。

## 1. 源代码生成/整理规则（强制）

### 1.1 明确无效降重手段（禁止）
- 仅改变量名、注释、空格、空行：禁止作为“降重”措施（可被识别，不算原创）。

### 1.2 代码比对粒度（生成时必须针对性规避）
按由粗到细的结构/语义层面：
- 整体文件结构：目录结构、文件命名、类/函数组织方式。
- 控制流/逻辑结构（CFG）：分支、循环、调用关系（改变量名无效）。
- 核心代码块：函数级、算法级，尤其常见模块（登录/权限/支付/加密/上传/分页等）。
- 细粒度特征：AST 结构、函数签名与参数列表、注释语义、开源指纹等。

### 1.3 必改 10 个结构级降重手法（必须落地，不得遗漏）
1. 重构函数拆分：长函数拆成 2-3 个子函数；合并零散小函数；打乱原有层级。
2. 调换执行顺序：同功能逻辑调换先后执行顺序（不改变最终结果）。
3. 重写判断条件：if/else 互换、三元互转、for/while 互换（保持语义一致）。
4. 重构传参方式：形参/实参组织调整；全局变量改局部；封装入参出参。
5. 新增私有工具方法：插入自用工具函数/校验器/格式化器等，形成独有代码向量。
6. 删减无用开源冗余：删框架示例、测试/演示代码、无用注释堆、脚手架残留。
7. 修改类结构：继承关系微调、接口实现顺序重排、拆分/组合服务层。
8. 数据处理逻辑微调：遍历/集合取值/排序/分页/过滤逻辑换写法。
9. 去除通用模板代码：登录、分页、文件上传等通用模块避免照搬，按业务重写逻辑。
10. 控制提交页内容：提交材料的前后重点页必须穿插自研业务代码，避免首尾全是框架。

### 1.4 风险阈值（内部红线，用于自检）
以下为“实务口径”落地成内部风控规则：
- 单段核心代码相似 > 15%：触发人工复核与解释准备。
- 整体核心代码相似 > 30%：高风险，视为可能驳回/要求独创性说明。
- 开源框架占比 > 40%：高风险，视为可能驳回（建议提交材料突出自研业务）。
- AI 生成代码特征明显：高风险，必须进行结构级重写与一致性证据补强。

### 1.5 提交代码材料卫生规范（必须）
- 剔除：外网地址、真实项目路径、作者信息、时间戳等可识别信息。
- 禁止：空文件、明显报错段、用注释堆砌“凑页数”。
- 禁止：首尾全是框架代码；必须嵌入自研业务逻辑。
- 行数建议：3000-6000 行最佳（工程经验；若提交口径另有要求则以提交口径为准）。

#### 1.5.1 源代码文档“说明性自然语言”控制（推荐）
问题：源代码文档通常会从 `apps/`、`packages/`、`scripts/` 收集源文件逐行编排，因此代码中的说明文字（例如“演示/软著/仅用于截图”等）也会进入源代码文档。

推荐做法：只在“源代码文档生成阶段”过滤，不强制改动业务代码本体：
1) 在项目根目录新增 `source_doc_filters.json`：
```json
{
  "strip_substrings": ["软著","演示","仅用于截图","文本种子","领域文本","数据结构","[padding]"]
}
```
2) 重新生成三件套（会重写源代码文档）：在 `{B}` 目录下执行：`python -X utf8 .\\scripts\\build_materials.py --name {A} --output-dir . --pdf-engine libreoffice`

说明：
- 过滤策略必须以“提交卫生”为目标，避免误删关键业务逻辑行。
- 若要求“源代码文档中不出现任何自然语言文案”，需要更激进的规则（例如仅保留代码语句行、剔除 HTML 文案/JS 文案/docstring 等），但会显著降低可读性与可核验性，需与审查口径权衡。

#### 1.5.2 源代码文档纯度门禁（强制，跨项目公共规则）
为避免“行业领域叙述/PRD 章节/运营文案”从 HTML/README 等渗入源代码文档，公共流水线已引入两道防线：
1) 生成阶段过滤：`scripts/build_materials.py` 默认过滤 `行业背景/用户角色/核心业务流程/数据模型/功能列表/边界约束/验收标准/视觉风格关键词/版本快照/发布留痕/资源清单/交互摘要/占位页面` 等叙述性关键词行。
2) 交付门禁：`scripts/postflight_baseline.py` 增加 `source_doc_purity` 检查，若 `{A}源代码文档.md` 命中上述关键词则直接判定 FAIL。

项目仍可按需新增 `source_doc_filters.json` 做更精细的补充过滤，但不得削弱上述公共门禁。

补充：严禁在源代码文档中出现可见占位符（例如 `[padding]`）。

补充：由 `generate_web_system.py` 生成的 `_auto_catalog.py` 必须使用当前需求的领域词汇命名条目标题与描述（例如“样本记录”），禁止再次使用“通用记录/通用条目”一类泛化文案；否则应视为模板污染并在生成器层修复。

同时，为避免出现大量空白页/空白行影响观感与审查稳定性，固定行数补齐策略升级为：
- 禁止使用“空行/不可见空白字符（例如 NBSP）”作为补齐手段；
- 若真实代码行不足 3000 行，使用“重复真实代码行片段（wrap-around）”补齐到 3000 行，且保证每一行非空（必要时用 `#` 占位注释行，而非空白字符）。

### 1.5.3 源代码文档过滤规则配置（强制，跨项目公共规则）
- 过滤与跳过规则的**唯一配置源**为 `E:\copyRight\workspace\shared\config\rules\source_doc_rules.json`。
- `scripts/build_materials.py` 只负责“按配置执行生成”，不得在代码中硬编码平行的关键词清单/跳过规则。
- 项目级微调仅允许在 `{A}/source_doc_filters.json` 追加 `strip_substrings/include_roots`，不得削弱公共门禁与公共规则。

### 1.5.4 操作手册规则配置（强制，跨项目公共规则）
- 操作手册的公共规则（字数阈值、截图数量阈值、尾部元信息剔除、补齐块内容）唯一配置源为 `E:\copyRight\workspace\shared\config\rules\manual_rules.json`。
- `scripts/build_materials.py` 仅负责“按配置执行生成”，不得在代码中硬编码平行的规则清单。

补充：操作手册 Markdown 正文允许包含多级目录编号（例如 `2.1`、`4.1.1`），并以 `#` 标题层级为准生成 docx 的目录与标题结构；编号应保持连续且不重复，避免 md 与 docx 出现双重编号。

## 2. 说明书/操作手册生成规则（强制）

### 2.0 字体与中英文字体映射（强制）
- Word 文档中：中文字符使用 `宋体`，非中文（ASCII/西文）使用 `Times New Roman`。
- 仅调整字体，不得改动字号、行距、分页规则与行号规则。
- 若生成器/后处理脚本会重写页眉页脚或封面字体，必须在最终输出阶段再次执行字体映射，确保一致。

### 2.0.1 PDF 生成链路（公共规范，强制）
- PDF 统一由 **LibreOffice headless 从 Docx 直接导出**（docx → pdf），作为公共默认链路。
- 禁止：md → html → 浏览器打印生成 pdf 作为默认链路（可保留历史能力但不得作为默认）。
- 生成顺序固定：先生成并后处理 docx（封面/页眉页脚/字体映射/分页清理）→ 再由 LibreOffice 导出 pdf。
- 注意：LibreOffice 的排版渲染与 Word 可能存在细微差异；以“公共规范一致性、可自动化稳定执行”为优先。
### 2.0.2 控制台/AI 一键生成（推荐命令）
在 `{A}`（需求目录）下执行：
- 正式交付：`python .\scripts\build_materials.py --name "<软件全名>" --output-dir . --pdf-engine libreoffice`
- 若必须在无桌面环境临时出 PDF（不推荐交付）：`python .\scripts\build_materials.py --name "<软件全名>" --output-dir . --allow-pdf-fallback`

补充：若仅需把现有 docx 转成 pdf（不重新生成 docx），在 `{A}` 下执行：
- `soffice.com --headless --nologo --nolockcheck --nodefault --nofirststartwizard --convert-to pdf --outdir . "<软件全名>操作手册.docx" "<软件全名>源代码文档.docx"`

### 2.1 文档比对粒度（生成时要主动差异化）
- 句子/段落级语义向量比对。
- 重点抓：功能描述、流程图、界面说明、专业术语搭配。
- AI 生成特征向量：句式过规整、无个性化细节、出现“幻觉描述”。

### 2.2 文档三大驳回点（内部验收即 Fail）
1. 模板套话、全网通用软著文案。
2. 明显 AI 生成：句式工整、无业务细节、空话多、逻辑断层。
3. 文档功能与源代码对不上（最常见失败原因之一）。

### 2.3 文档降重改写标准（必须全部执行）
1. 删除通用万能话术（如“界面简洁、操作简单、功能强大”等）。
2. 全部替换为项目专属业务描述：写场景、流程、字段、规则、异常分支。
3. 段落语序重排，句式改写，避免与常见模板同结构。
4. 流程图重新绘制，不套用网上通用流程图。
5. 界面截图说明要标注：独有按钮/字段/模块名，并与代码/配置能对应。
6. 功能描述严格对应代码实现：多一项/少一项都不行。

### 2.4 去 AI 文风快速规则（必须）
- 减少长排比句和“首先/其次/最后/综上所述”等模板连接词堆叠。
- 增加真实业务细节与少量项目专属术语（且必须可在代码或配置中定位）。
- 禁止虚构第三方系统对接、权限体系、指标口径、页面/按钮等不存在内容。

### 2.5 Web 系统页面截图样式约束（强制）
用于约束“操作手册/说明书”的界面截图与对应文字说明，避免多需求截图同质化、缺少可验证交互要素。

#### 2.5.1 页面布局差异化（按需求独立）
- 每个需求（或子系统/模块）的页面布局必须独立成套：导航结构、筛选区、列表/卡片区、详情区、字段组合、信息层级至少满足其一存在明显差异。
- 不同需求之间必须体现“行业区分”：页面模块命名、字段/指标口径、图表类型、列表列项、状态流转用词等，至少 3 处以上体现行业特征（且能在代码路由/菜单/配置或数据字典中定位）。
- 禁止：多个需求复用同一套页面骨架仅替换标题/菜单名/少量字段；禁止所有页面都采用同一类“通用后台模板布局”导致审查侧观感同质化。

#### 2.5.2 操作按钮与弹窗（必须展示可执行交互）
- 页面截图必须包含“操作入口”：至少展示一次带按钮或行内操作的列表页/详情页，按钮应覆盖 `新增 / 编辑 / 删除 / 查看` 中的全部或与该模块一致的等价动作（如“创建/修改/移除/详情”），并在截图说明中明确按钮位置与触发结果。
- 必须包含弹窗截图：对存在弹窗交互的模块，至少给出 2 张弹窗截图（如“新增弹窗”“编辑弹窗/查看弹窗”“删除确认弹窗”中的任意两类），且弹窗内要出现可核验的信息要素（表单字段、必填标识、校验提示、按钮文案等）。
- 禁止：只给静态首页/看板页截图、不展示任何可操作按钮；或仅描述“可新增/编辑/删除”但截图中看不到入口与弹窗。

#### 2.5.3 截图说明的最小信息量（与代码一致性）
- 每张截图说明至少包含：页面/弹窗名称、主要字段（≥3 个）、关键按钮（≥1 个）、至少 1 条业务规则或校验提示（例如必填、范围、唯一性、状态限制等）。
- 截图对应的页面/弹窗名称、按钮文案、字段名必须与代码/配置一致（路由、菜单、表单 schema、国际化文案或字典项可定位其一即可）。

#### 2.5.4 操作手册模板骨架与“图片+图题”规则（强制，跨项目公共规则）
为避免操作手册内容“自写漂移”导致与 skill 模板差异过大，公共流水线在 `scripts/manual_style_validator.py` 引入模板骨架门禁：
1) `{A}操作手册.md` 必须包含模板关键章节标题（允许章节编号前置插入“文档定位与适用对象”，但模板章节标题不得缺失；“一级功能模块与二级菜单操作”允许简化为“一级功能模块”）。
2) 每一张 `images/*.png` 引用后，必须在附近出现图题说明行，格式形如：`图X-Y 描述`（支持 `图3-2a`）。
3) 仍需同时满足按钮文案与交互说明门禁：`新增/编辑/删除/查看` 以及至少 2 处弹窗/确认类交互描述。

建议：以 `chinese-copyright-application/references/operation-manual-template.md` 为写作基线填充，再由 `build_materials.py` 完成封面、页眉页码与 docx/pdf 渲染。

## 3. 生成过程的“强约束提示词模板”（可直接供 skill 调用）

将以下约束作为系统级生成规则嵌入生成器（对代码与文档均适用）：
- 只允许输出“可定位到代码/配置/截图”的内容；无法定位的一律不输出。
- 每个功能点必须带“可定位证据”：代码路径 + 类/函数/接口/路由名（至少一项）。
- 禁止通用套话；每段必须包含至少 1 个项目专属字段/规则/异常提示信息。
- 禁止用变量名/注释替换来充当原创性；必须做结构级重构。
- 操作手册截图必须满足：不同需求页面布局显著差异化，且包含可见操作按钮与对应弹窗（新增/编辑/删除/查看等）。
- 截图说明必须包含：页面/弹窗名称、字段清单、按钮文案、至少 1 条业务规则/校验提示，并与代码/配置可定位实体一致。
