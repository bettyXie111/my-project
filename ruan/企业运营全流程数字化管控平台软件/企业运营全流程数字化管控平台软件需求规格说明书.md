# 企业运营全流程数字化管控平台软件需求规格说明书

## 1. 概要
- 产品/功能名称：企业运营全流程数字化管控平台软件
- 一句话描述：面向集团型企业经营场景的全流程数字化管控平台，覆盖主数据、采购、库存、销售、合同、预算、报销、审批、审计与通知闭环。
- 责任团队：自动化交付代理
- 日期/版本：2026-05-13 / V1.0
- 交付场景：新建独立交付项目

## 2. 问题定义
传统企业运营系统常见的问题是业务链条被拆散在多个表格、邮件和聊天工具之中，采购申请看不到预算余额，合同审批与回款登记分离，库存收货无法及时反馈到经营看板，导致管理者只能事后追责而不是事前控制。本项目需要提供一个统一入口，把“组织与主数据维护、业务单据发起、预算控制、审批流转、经营看板和审计追踪”整合到同一套平台中，使集团总部、区域运营、采购、财务、仓储和审计岗位共享同一套实时状态。

## 3. 目标与成功标准
- 业务目标：
  1. 将采购、销售、财务、审批和审计数据统一在单一平台内管理。
  2. 在单据提交时完成预算前置校验，减少超预算和重复付款。
  3. 为管理层提供可追溯的经营看板和审计留痕。
- 用户结果：
  1. 业务人员能够在 3 分钟内完成一张标准业务单据录入。
  2. 审批人能够在待办中心直接完成审批动作并查看流转结果。
  3. 财务和审计人员能够按请求号或业务编号追溯关键操作。
- 成功指标：
  1. 自动化集成测试全部通过。
  2. 平台代码量不少于 3000 行；当前源码统计为 7752 行。
  3. 采购申请、合同审批、报销付款三条主流程均可在本地种子数据下完成闭环演示。

## 4. 范围与非目标
- 范围内：
  1. 统一登录、令牌刷新、退出登录、当前用户信息。
  2. 组织、岗位、角色、权限、成本中心、客户、供应商、物料、项目、仓库等主数据。
  3. 采购申请、采购订单、收货入库、库存余额。
  4. 销售订单、合同审批、回款登记。
  5. 预算、费用报销、付款申请。
  6. 流程模板、待办、已办、我的发起、通知、审计日志、文件元数据占位接口。
- 非目标：
  1. 不包含复杂 BI 报表设计器。
  2. 不包含第三方支付、电子签章和短信网关的真实对接。
  3. 不包含多租户 SaaS 计费系统。

## 5. 用户与核心用例
- 主要用户：
  1. 平台管理员
  2. 运营总监
  3. 销售专员
  4. 采购专员
  5. 财务专员
  6. 仓储专员
  7. 审计专员
- 次要用户：
  1. 总部经营管理层
  2. 区域负责人
- 用例 1：销售人员创建销售订单，管理员发起合同审批，财务登记回款，首页看板同步更新。
- 用例 2：采购人员发起采购申请，预算校验通过后进入审批，审批通过后生成采购订单并完成入库。
- 用例 3：业务人员提交报销，财务创建付款申请，审计人员通过日志和通知进行追踪。

## 6. 功能需求
1. 系统必须提供账号密码登录、访问令牌刷新、退出登录和当前用户信息查询能力。
2. 系统必须对接口执行基于角色权限和数据范围的访问控制，未经授权的请求返回统一错误码。
3. 系统必须提供客户、供应商、物料、项目、仓库和主数据汇总查询接口。
4. 系统必须支持快速创建客户，并在主数据页面实时回显新增结果。
5. 系统必须支持发起采购申请，并在提交时校验预算键、申请组织、成本中心和物料明细。
6. 系统必须支持对审批通过的采购申请生成采购订单。
7. 系统必须支持基于采购订单执行收货入库，并更新库存余额。
8. 系统必须支持创建销售订单，并基于销售订单发起合同审批。
9. 系统必须支持合同审批完成后登记回款并更新合同回款进度。
10. 系统必须支持新增预算、发起费用报销、创建付款申请。
11. 系统必须在预算不足时阻断报销或采购提交，并返回业务级冲突提示。
12. 系统必须支持流程模板查看、待办审批、已办查看和我的发起列表。
13. 系统必须提供经营看板，展示销售、采购、库存、费用和通知 KPI。
14. 系统必须提供审计日志与通知中心，并支持站内通知设为已读。
15. 系统必须提供集成测试，覆盖登录、权限、采购入库、合同回款、预算拦截和付款幂等等关键路径。

## 7. 交互与体验要求
- 关键页面/流程：
  1. 登录页
  2. 首页经营看板
  3. 主数据页面
  4. 采购与库存页面
  5. 销售与合同页面
  6. 费用与预算页面
  7. 流程待办页面
  8. 审计与通知页面
- 状态处理：
  1. 成功提交后使用页面内提示条反馈结果。
  2. 权限不足、预算不足、单据冲突等异常使用统一错误提示。
  3. 空列表场景显示“暂无数据”或“暂无待办”。
- 可访问性/本地化：
  1. 页面文案统一中文。
  2. 颜色系统使用明确品牌色与对比色，避免默认紫色模板风格。
  3. 在 1024px 以下屏幕下可切换为单列布局。

## 8. 技术栈
- 前端：原生 HTML + CSS + ES Modules JavaScript
- 后端：Python 3.11 + WSGI 应用工厂
- 数据库/存储：SQLite 3
- 认证/身份：自签名令牌 + Refresh Token + RBAC + 数据范围控制
- 基础设施/部署：本地单进程部署，可迁移到 Windows/Linux 服务器
- CI/CD：当前以脚本触发为主，保留接入 GitHub Actions 的结构化入口
- 测试栈：Python unittest 集成测试 + Playwright 页面截图验证
- 假设：当前离线交付环境优先保证可运行与可验证，因此未引入 Next.js 14、Prisma 和 PostgreSQL；后续若上云，可沿现有模块边界迁移。

## 9. 系统架构
- 高层架构：
  1. 浏览器加载静态前端页面。
  2. 前端通过 `/api/v1` 调用 Python WSGI 后端。
  3. 后端通过 SQLite 持久化业务数据，并在本地文件中运行迁移与种子数据。
- 模块边界：
  1. `core` 负责配置、数据库、路由、响应和安全。
  2. `modules` 负责认证、主数据、采购、库存、销售、财务、流程、通知、审计等领域逻辑。
  3. `web/src/views` 按页面拆分渲染逻辑。
  4. `packages/shared-contracts` 保存接口契约说明。
- 请求/数据流：
  1. 用户在前端发起动作。
  2. `api.js` 统一封装请求、令牌刷新和幂等头。
  3. 路由层解析请求并创建上下文。
  4. 领域模块执行业务逻辑并返回统一 JSON 响应。
- 第三方服务：
  1. 当前版本不依赖外部 SaaS 服务。
  2. 浏览器截图和 PDF 导出依赖本地 Edge + Playwright 工具链。

## 10. 数据模型与存储设计
- 核心实体：
  1. 用户、角色、组织、岗位、成本中心、系统配置
  2. 客户、供应商、物料、项目、仓库
  3. 预算、采购申请、采购订单、库存流水、库存余额
  4. 销售订单、合同、回款记录
  5. 费用报销、付款申请
  6. 工作流模板、工作流实例、审批任务、通知、审计日志、文件资产
- 关系与索引策略：
  1. 采购申请与采购订单一对多。
  2. 合同与销售订单一对一关联，回款记录与合同一对多。
  3. 预算通过 `budget_key` 连接业务单据。
  4. 关键表在业务类型、状态、时间字段上建立索引。
- 生命周期/保留策略：
  1. 主数据默认长期保留并使用状态字段控制启停。
  2. 审计日志和通知记录按业务追溯保留。
- 迁移与回填：
  1. 启动时自动执行 SQL 迁移。
  2. 本地开发和测试使用种子脚本重置样例数据。

## 11. API 与接口契约
- API 风格：REST 风格 JSON API
- 端点定义：
  1. 认证：`POST /auth/login`、`POST /auth/refresh`、`GET /users/me`
  2. 主数据：`GET /masterdata/summary`、客户/供应商/物料/项目/仓库查询与客户创建
  3. 采购：`GET/POST /procurement/requests`、`GET/POST /procurement/orders`
  4. 库存：`GET /inventory/balances`、`POST /inventory/receipts`
  5. 销售：`GET/POST /sales/orders`、`GET/POST /contracts`
  6. 财务：预算、报销、付款申请、回款接口
  7. 流程：模板、待办、审批动作
  8. 审计：`GET /audit-logs`、通知列表与已读
- 请求/响应模式：
  1. 成功返回 `code=OK`、`message=success`、`data`、`requestId`。
  2. 失败返回统一业务错误码，如 `AUTH_403_01`、`BUDGET_409_01`。
- 错误与重试：
  1. 幂等性要求作用于采购申请、入库、付款申请等重复提交风险较高的接口。
  2. 客户端收到 401 时自动尝试刷新令牌并重试原请求一次。
- 版本兼容：
  1. 所有接口统一挂载在 `/api/v1`。

## 12. 安全、隐私与合规
- 威胁/滥用场景：
  1. 越权访问审计日志或财务数据。
  2. 重复提交付款和入库。
  3. 使用失效令牌访问接口。
- 安全控制：
  1. 认证、权限点校验、数据范围控制。
  2. 幂等键控制重复创建。
  3. 统一错误响应避免泄露内部堆栈。
- 隐私与 PII：
  1. 样例数据仅包含演示账号与示例联系方式。
  2. 生产化时应将真实手机号、邮箱与证件信息替换为受控数据。
- 合规约束：
  1. 审计日志需记录关键动作、业务类型、请求号和变更摘要。

## 13. 非功能要求
- 性能：
  1. 本地单机环境下常规查询接口目标响应时间 p95 小于 300ms。
  2. 复杂提交流程目标响应时间 p95 小于 800ms。
- 可靠性：
  1. 启动即迁移和种子校验，保证本地环境可重复运行。
  2. 自动化测试通过后方可交付。
- 可扩展性：
  1. 模块按领域拆分，便于后续迁移数据库和部署方式。
- 成本：
  1. 当前版本不依赖付费云服务和商业中间件。

## 14. 可观测性与运维
- 日志/指标/追踪：
  1. 后端为每次请求返回 `requestId`。
  2. 审计日志记录关键业务动作。
- 告警与运行手册：
  1. 预算不足、权限不足、重复提交等业务错误必须直接反馈到前端。
  2. 通知中心展示需要用户关注的事件。
- 特性开关与回滚：
  1. 配置项支持预算控制模式与附件大小限制。
