# 中国软件著作权申请材料生成

> 更新日期：2026-05-15
> 本文件侧重软著三件套的格式规范、字段规则和一致性约束；流程编排、阶段顺序和自动化执行门槛以 `automation_prompt.md` 为准。

## 规则优先级
1. 用户当前会话的最新明确要求 > 本文件 > 历史默认规则。
2. 同一主题冲突时，以本文件最新条款为准。
3. 无法从代码或文档可靠推断的主体信息不得编造。

## 输入参数
- 需求名称：`{A}`（必填，作为目录名与文件名前缀）
- 项目根目录：`{PROJECT_ROOT}`（默认当前目录）
- 需求文档路径：`{REQUIREMENT_PATH}`（可选）

## 目录与命名（强制）
必须在 `{PROJECT_ROOT}` 下创建并使用独立目录：
- `{PROJECT_ROOT}/{A}/`

所有产物必须落在该目录，不使用固定 `copyright-application-materials/` 目录。

## 必交付产物
至少输出以下文件：
- `{A}/{A}需求规格说明书.md`（若已由上游 skill 生成则直接复用）
- `{A}/{A}操作手册.md`
- `{A}/{A}操作手册.docx`
- `{A}/{A}操作手册.pdf`
- `{A}/{A}源代码文档.md`
- `{A}/{A}源代码文档.docx`
- `{A}/{A}源代码文档.pdf`
- `{A}/软件著作权登记申请表.md`
- `{A}/软件著作权登记申请表.docx`
- `{A}/images/`（操作手册截图）

## 通用规则（三份文档共用）

### 1. 页眉与页码
- 操作手册、源代码文档、申请表的每页都必须有页眉。
- 页眉内容固定为：`软件全称 + 版本号`，中间不得有空格，版本号中的 `V` 必须大写。
- 每页底部必须有连续页码，页码从第 1 页开始。

### 2. 内容一致性
- 三份文档中的软件全称、版本号必须完全一致。
- 操作手册描述的功能不得超出源代码实现范围。
- 源代码中的业务模块必须与操作手册、申请表中的功能描述一一对应。
- 编程语言、代码总行数在申请表和源代码文档中必须一致。
- 操作手册中的截图必须与实际运行系统界面一致。

### 3. 去 AI 化与差异化
- 避免 AI 套话（如“首先其次最后”“综上所述”等）。
- 同一项目三份文档语气风格统一，不同项目允许风格差异。
- 变量、函数、模块命名必须业务相关，避免泛化命名。
- 文档内容与截图应通过 `ai-slop-detector` 检查。

### 4. UI 与截图差异化
- 不同项目应在配色、布局、字体、圆角、图标上体现差异。
- 手册截图必须体现该项目真实风格，不复用模板图。

## 申请表规范

### 基本信息
- 软件全称：与操作手册、源代码文档一致。
- 版本号：`V1.0`（`V` 大写）。
- 软件分类：`应用软件`。
- 源程序量：与源代码文档总行数一致（例如 `3000行`）。

### 主体信息与日期
以下信息按实际填写；无法推断时允许留空，不得编造：
- 著作权人名称
- 统一社会信用代码/证件号码
- 联系人
- 电话
- 地址
- 邮编
- 邮箱
- 开发完成日期（`YYYY-MM-DD`）
- 首次发表日期（未发表可留空）

## 操作手册规范

### 1. 封面
- 操作手册封面统一为双行居中大标题：第一行 `软件全称 + 空格 + V1.0`，第二行 `操作手册`。
- 两行均为宋体 24。
- 封面主标题与副标题之间不得插入空行，双行作为一个整体在封面页内居中显示。
- 封面后分页进入正文，正文首页首行不得再出现“操作手册”副标题。
- 正文首页首行不得为空段落，必须直接进入正文标题或正文内容。

### 2. 章节组织
- 建议覆盖：系统入口与登录、主要功能模块、端到端业务流程、常见问题。
- “文档说明”归属第一章节“文档定位与适用对象”，不得归入第二章节“系统入口与登录”。

### 3. 截图规范
- 必须使用真实运行界面截图，不得使用占位图。
- 截图下方必须有图题，格式：`图X-Y 描述`。
- 截图应包含关键导航与页面主体，文字清晰可读。

### 4. 字数要求
- 默认不少于 3000 字；若上游任务设置更高阈值，按更高阈值执行。

## 源代码文档规范

### 1. 基础格式
- 使用 Word 文档。
- 采用等宽字体（如 `Consolas` / `Courier New`），字号 10 或 11。
- 代码必须以纯文本嵌入，不得用图片/文本框/表格承载代码。
- 源代码文档不要求任何行号展示（不使用 Word 原生行号，也不在正文写手工行号）。

### 2. 空行与换行
- 禁止无意义视觉换行；长行必须按断行规则拆分为独立代码行。
- 长行必须在语法断点手动断行。
- 每页固定 50 行，页末最后一行必须为非空内容行。

### 3. 总行数与分页
- 总行数固定 3000 行，共 60 页。

### 4. 代码流采样规则（强制）
- 先按稳定顺序拼接代码流：入口 -> 路由/页面 -> 业务模块 -> 配置工具 -> 路径字典序。
- 总行数 >3000：取整体代码流前 1500 行 + 后 1500 行。
- 总行数 <=3000：全部收录。
## 生成与转换要求

### 信息收集
优先读取：
- 微信小程序：`app.json`、`project.config.json`、`package.json`、`README.md`
- Web/Node：`package.json`、`README.md`、路由/菜单/接口目录
- 其他：`pom.xml`、`build.gradle`、`Cargo.toml`、`.csproj`、`pyproject.toml`、`requirements.txt`

### 内置工具优先级
1. `scripts/ensure_pandoc.ps1`
2. `scripts/capture_screenshots.mjs`
3. `scripts/md_to_docx_with_images.py`
4. `scripts/generate_copyright_docs.py`（仅可用于草稿提取）

### Markdown 转 Docx/PDF
1. 优先使用 `pandoc` 转换。
2. 操作手册优先使用 `references/operation-manual-template.docx`。
3. 带图文档要确保图片真实嵌入。
4. 若系统无 `pandoc`，先执行 `scripts/ensure_pandoc.ps1`。

## 输出质量门槛
1. `{A}` 目录与 md/docx/pdf/images 是否齐全。
2. 申请表字段是否完整、字数是否合规。
3. 三份文档的软件全称、版本号、编程语言、代码总行数是否一致。
4. 操作手册截图是否真实、完整、清晰。
5. 源代码文档是否满足 3000 行、60 页、每页 50 行。
6. 文档文本是否完成去 AI 化。

## 完成后必须输出
1. 自动化测试结果（命令、通过/失败、关键摘要）。
2. 代码总行数统计（>=3000）。
3. 操作手册字数统计（按当前阈值）。
4. 源代码文档总行数与分页统计。
5. 三份文档生成状态（md/docx/pdf）。
6. 最终文件绝对路径清单。
7. 若失败则自动重试一次；仍失败则给出明确阻塞点。

## 本 skill 阶段完成判定

- 判定条件：
  - `{A}/{A}需求规格说明书.md` 已存在且为完整 PRD（非骨架）。
