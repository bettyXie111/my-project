# 多维度绩效评价与智能分析系统需求规格说明书

## 1. Summary
- Product/Feature Name: 多维度绩效评价与智能分析系统
- One-line Description: 面向企业 HR、直属经理与员工的绩效计划、评价执行、结果分析与改进闭环平台。
- Owner/Team: 产品与交付联合小组
- Date/Version: 2026-05-13 / V1.0
- Delivery Context: 新建系统

## 2. Problem Statement
现有绩效管理流程普遍依赖 Excel、邮件与即时通信工具分散协作，存在指标口径不统一、审批链路不可追踪、员工自评与经理评价分离、结果分析滞后、低绩效改进动作闭环弱等问题。企业需要一套可审计、可追踪、可量化的绩效平台，把绩效计划、审批、自评、经理打分、风险识别与改进动作统一在同一系统中。

## 3. Goals and Success Criteria
- Business Goals:
  - 建立统一的绩效计划与指标库。
  - 把绩效审批、自评、经理评价、分析预警和改进动作纳入一条闭环。
  - 形成可直接用于审计和管理复盘的数据底座。
- User Outcomes:
  - HRBP 能快速发起绩效周期与绩效计划。
  - 员工能在截止日前完成自评并查看结果。
  - 直属经理能在线审批计划并完成评分。
  - 管理层能查看低分预警、部门排名和改进进度。
- Success Metrics:
  - 绩效计划创建到审批完成全链路可在线追踪。
  - 经理评价完成后自动计算总分与等级。
  - 低分计划可在系统内生成改进动作并跟踪状态。
  - 首页可展示待办、平均分、低分人数、改进动作等关键指标。

## 4. Scope and Non-Goals
- In Scope:
  - 登录鉴权、角色权限、组织与用户数据复用。
  - 绩效指标库管理。
  - 绩效周期管理。
  - 绩效计划创建、提交审批、审批流转。
  - 员工自评、经理评价、自动算分与等级判定。
  - 智能分析看板、低分预警、部门排名。
  - 改进动作创建与提醒。
  - 审计日志、站内通知。
- Out of Scope:
  - 薪酬发放。
  - 第三方 IM/邮件真实集成。
  - 多租户隔离。
  - 复杂 BI 自助分析器。

## 5. Users and Core Use Cases
- Primary users:
  - 系统管理员
  - HRBP/绩效专员
  - 直属经理
  - 普通员工
- Secondary users:
  - 绩效委员会负责人
  - 审计人员
- Use Case 1:
  - HRBP 创建绩效周期，配置指标库，并为员工发起绩效计划。
- Use Case 2:
  - 绩效计划经过负责人和直属经理审批后，员工提交自评，经理完成打分。
- Use Case 3:
  - 管理层查看平均分、低分名单和改进动作，并分派后续辅导任务。

## 6. Functional Requirements
1. 系统必须支持用户名密码登录、令牌续期、退出登录和当前用户信息查询。
2. 系统必须支持按角色控制主数据维护、计划审批、自评、经理评价、分析查看与审计查看权限。
3. 系统必须支持维护绩效指标库，指标至少包含编码、名称、维度、权重和评分规则。
4. 系统必须支持维护绩效周期，周期至少包含编码、名称、周期类型、开始日期、结束日期、自评截止日和经理评价截止日。
5. 系统必须支持为指定员工创建绩效计划，计划至少包含周期、员工、直属经理、所属组织、标题和指标卡。
6. 系统必须支持提交绩效计划审批，并自动创建审批实例与审批任务。
7. 系统必须支持两级审批链路：负责人审批、直属经理审批。
8. 系统必须在绩效计划审批通过后允许员工提交自评。
9. 系统必须在员工提交自评后允许直属经理提交经理评价。
10. 系统必须对经理评价的每项指标分数进行合法性校验，分值范围为 0-100。
11. 系统必须按指标权重自动计算绩效总分并生成等级。
12. 当绩效总分低于阈值 75 时，系统必须能够创建改进动作。
13. 系统必须提供分析视图，展示绩效计划总量、待自评、待经理评价、平均分、低分人数、部门排名、等级分布和改进动作。
14. 系统必须提供站内通知，通知审批、评价完成和改进动作分派。
15. 系统必须记录登录、计划创建、自评提交、经理评价、改进动作创建等审计日志。

## 7. UX and Interaction Requirements
- Key screens/flows:
  - 登录页
  - 绩效看板
  - 组织与指标库
  - 绩效计划
  - 评价执行
  - 分析与改进
  - 审批中心
  - 审计与通知
- State handling:
  - 每个列表页必须具备空态文案。
  - 表单提交必须反馈成功或失败提示。
  - 审批、自评、经理评价失败时必须返回明确错误消息。
- Accessibility/localization requirements:
  - 全站中文界面。
  - 所有按钮与输入框保留文本标签。
  - 支持桌面浏览器访问。

## 8. Technical Stack (Must Be Explicit)
- Frontend: 原生 HTML + CSS + ES Modules 单页应用
- Backend: Python 3 标准库 WSGI 服务
- Database/Storage: SQLite
- Auth/Identity: 本地用户表 + 签名 Token
- Infrastructure/Deployment: 单机运行，静态前端由同一 WSGI 服务托管
- CI/CD: 本地脚本驱动
- Testing stack (unit/integration/e2e): Python `unittest` 集成测试，Playwright 截图验证
- Recommended Stack Option A (+ tradeoffs):
  - 当前实现。优点是零额外依赖、易离线交付；缺点是扩展性与工程化能力弱于大型框架。
- Recommended Stack Option B (+ tradeoffs):
  - React + FastAPI + PostgreSQL。优点是更易扩展；缺点是当前环境额外依赖更重，不适合本次离线快速交付。
- Pending confirmation:
  - 本次交付按方案 A 实施。

## 9. System Architecture
- High-level architecture:
  - 单体 Web 应用，前后端同仓库，WSGI 同时提供 API 与静态页面。
- Module/service boundaries:
  - `auth`: 登录、刷新、登出、当前用户
  - `iam`: 组织、角色、用户、成本中心、系统配置
  - `masterdata`: 绩效指标与绩效周期
  - `workflows`: 审批模板、审批任务、审批动作
  - `procurement`（已重构为绩效计划域）: 绩效计划创建与提交审批
  - `sales`（已重构为评价执行域）: 员工自评与经理评价
  - `finance`（已重构为分析改进域）: 智能分析与改进动作
  - `dashboard`: 首页汇总看板
  - `notifications` / `audit`: 通知与审计
- Request/data flow:
  - 主数据维护 -> 绩效计划创建 -> 工作流审批 -> 员工自评 -> 经理评价 -> 智能分析 -> 改进动作
- Third-party services and responsibilities:
  - 无强依赖第三方在线服务。

## 10. Data Model and Storage Design
- Core entities and fields:
  - `performance_indicators`: 指标编码、名称、维度、权重、评分规则
  - `performance_cycles`: 周期编码、名称、期间、截止日
  - `performance_plans`: 计划编号、周期、员工、经理、指标卡、自评、经理评价、总分、等级、状态
  - `improvement_actions`: 改进动作编号、所属计划、责任人、发起人、到期日、标题、详情、状态
- Relationships/indexing strategy:
  - 一个周期对应多个绩效计划。
  - 一个绩效计划对应一个员工和一个直属经理。
  - 一个绩效计划可关联多个改进动作。
  - 对周期、状态、员工、经理、到期日建立查询索引。
- Data lifecycle/retention:
  - 绩效计划和改进动作长期保留，用于审计和历史分析。
- Migration/backfill requirements:
  - 通过新增 SQL migration 创建绩效域表，无历史回填要求。

## 11. API and Interface Contracts
- API style: REST
- Endpoint/interface definitions:
  - `GET /api/v1/dashboard/performance`
  - `GET/POST /api/v1/performance/indicators`
  - `GET/POST /api/v1/performance/cycles`
  - `GET/POST /api/v1/performance/plans`
  - `POST /api/v1/performance/plans/{planId}/submit`
  - `POST /api/v1/performance/plans/{planId}/self-review`
  - `POST /api/v1/performance/plans/{planId}/manager-review`
  - `GET /api/v1/performance/analytics/overview`
  - `GET/POST /api/v1/performance/improvement-actions`
  - `GET /api/v1/approvals/tasks`
  - `POST /api/v1/approvals/tasks/{taskId}/actions`
- Request/response schemas:
  - 统一 JSON 包装结构：`code`、`message`、`requestId`、`data`
  - 列表接口统一返回 `items/page/pageSize/total`
- Error codes and retry behavior:
  - 认证失败 `AUTH_401_01`
  - 权限不足 `AUTH_403_01`
  - 状态冲突使用 `409`
  - 校验失败使用 `400`
- Versioning/backward compatibility:
  - API 统一挂载在 `/api/v1`

## 12. Security, Privacy, and Compliance
- Threats/abuse cases:
  - 未授权人员查看他人绩效结果
  - 非直属经理提交评分
  - 计划重复创建
- Security controls:
  - 基于角色的接口权限校验
  - 关键动作审计
  - 自评、经理评价、审批均校验执行人身份与计划状态
- Privacy/PII handling:
  - 用户邮箱和手机号在用户信息接口中做脱敏展示
- Compliance constraints:
  - 满足内部审计留痕要求
